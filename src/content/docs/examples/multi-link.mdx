Each requirement is backed by two tests: a standalone function and a class-scoped method. Both must pass for the requirement to be marked verified.

## The Spec

```markdown
---
title: User Service
modules:
  - src/
---
# User Service
CRUD operations for user management.
## Requirements
- [ ] Create user stores name and email [[ tests/test_user_service.py:test_create_user, tests/test_user_service.py:TestUserService::test_create_stores_fields ]]
- [ ] Get user retrieves by ID [[ tests/test_user_service.py:test_get_user, tests/test_user_service.py:TestUserService::test_get_returns_user ]]
- [ ] Update user changes stored fields [[ tests/test_user_service.py:test_update_user, tests/test_user_service.py:TestUserService::test_update_changes_email ]]
- [ ] Delete user removes from store [[ tests/test_user_service.py:test_delete_user, tests/test_user_service.py:TestUserService::test_delete_is_permanent ]]
```

## Run It

```sh
sodide verify examples/05-multi-link
```

## What to Notice

<Aside type="note" title="Comma-separated links for multiple test references">
Multiple verifiers are listed inside a single `[[ ]]` pair, separated by commas. sodide runs all of them and marks the requirement `✓` only when every linked test passes. If any one test fails, the requirement is `✗`. This is useful when a feature has both a happy-path test and a more thorough class-based test suite that should both be green before the requirement is considered done.
</Aside>

<Aside type="note" title="Class::method node ID syntax">
`TestUserService::test_create_stores_fields` refers to the `test_create_stores_fields` method inside the `TestUserService` class. The `Class::method` format is the node ID syntax used by pytest (and understood by sodide). Use it to pin a specific test method within a test class rather than running the entire class.
</Aside>

<Aside type="note" title="CRUD as a coverage pattern">
Pairing a standalone function test with a class-scoped method for each CRUD operation gives two independent angles on the same behavior: one that tests the function directly, and one that tests it through a fixture-driven test class. If both pass, the implementation is exercised from multiple call sites — a useful baseline for service-layer code.
</Aside>
